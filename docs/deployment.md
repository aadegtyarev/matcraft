# Развёртывание matcraft

## Целевая аудитория

- **Пользователи** — скачивают готовый бинарник под свою ОС со страницы GitHub
  Releases и запускают его. Rust и сборка из исходников не требуются.
- **Разработчики** — собирают из исходников (`cargo install --path .`, см. README,
  раздел «Для разработчиков»).

## Канал распространения

**GitHub Releases** — готовые бинарники под четыре платформы, собираемые CI:
<https://github.com/aadegtyarev/matcraft/releases>.

crates.io — **не текущий** канал: крейт не опубликован, `cargo install matcraft` не
работает. Публикация в crates.io возможна в будущем как дополнительный канал, но
сейчас не используется (crates.io-токен не нужен).

## Процесс релиза (для мейнтейнера)

Бинарники собирает workflow `.github/workflows/release.yml`. Запускается двумя путями:

### 1. Релиз с изменением крейта

Когда меняется сам крейт (код, зависимости, версия):

1. Bump `Cargo.toml` + `Cargo.lock`, запись в CHANGELOG, merge в `main`.
2. `git tag vX.Y.Z`
3. `git push --tags`
4. Push тега `v*` запускает `release.yml` → сборка 4 таргетов → создание GitHub
   Release + прикрепление архивов и их sha256.

### 2. Ручной запуск / backfill

Собрать бинарники для уже существующего тега (например, первая сборка после
внедрения workflow):

Actions → workflow **release** → **Run workflow** → выбрать существующий тег как ref
(`workflow_dispatch`).

Workflow должен уже находиться в ветке по умолчанию (`main`), чтобы появиться в
списке диспетчеризации Actions — поэтому первые бинарники получают ручным запуском
на существующем теге **после** merge workflow в `main`.

Job `create-release` идемпотентен (`gh release create … || true`): повторный запуск
на уже выпущенном теге не падает, а загрузка ассетов перезаписывает их начисто.

## Целевые платформы (4)

| Таргет | Раннер | Примечание |
|---|---|---|
| `x86_64-unknown-linux-musl` | `ubuntu-latest` | Полностью статический (musl); ставится `musl-tools` |
| `x86_64-pc-windows-msvc` | `windows-latest` | Нативная сборка; бинарник `matcraft.exe`, архив `.zip` |
| `x86_64-apple-darwin` | `macos-latest` | Кросс-компиляция с arm64-раннера |
| `aarch64-apple-darwin` | `macos-latest` | Нативная сборка на arm64-раннере |

`macos-latest` теперь Apple Silicon (arm64): `aarch64-apple-darwin` собирается
нативно, `x86_64-apple-darwin` — кросс-компиляцией (clang от Apple умеет это без
внешнего линкера). Тесты в release-workflow не гоняются — за тесты отвечает `ci.yml`;
здесь только сборка, архивирование и загрузка.

## Артефакты

- Именование: `matcraft-<tag>-<target>.{tar.gz,zip}` — например
  `matcraft-v0.9.1-x86_64-unknown-linux-musl.tar.gz`,
  `matcraft-v0.9.1-x86_64-pc-windows-msvc.zip`. `<tag>` = имя тега (с префиксом `v`).
- Формат: `.tar.gz` на Linux/macOS, `.zip` на Windows.
- Рядом с каждым архивом — файл `.sha256` с контрольной суммой.

## Безопасность и секреты

- Единственный токен — **эфемерный `GITHUB_TOKEN`**, который GitHub выдаёт раннеру на
  время запуска. Это не хранимый секрет, он не коммитится. Область прав ограничена
  `contents: write` на уровне job (least privilege), не на весь workflow.
- crates.io-токен **не требуется** — крейт не публикуется.
- Бинарники **не подписаны** (нет платных сертификатов для подписи/нотаризации).
  Следствие для пользователя — блокировки Gatekeeper (macOS) и SmartScreen (Windows);
  обход задокументирован в README (разделы macOS и Windows).

## Ошибки/поддержка

- **Баги и проблемы:** GitHub Issues
- **Обратная связь:** GitHub Issues
- **Канал распространения:** GitHub Releases
- **Мониторинг:** отсутствует (хобби-проект, без SLA)

### Частичный набор ассетов (fail-state)

Матрица сборки не fail-fast: если одна нога падает, падает только она, остальные
загружают свои архивы. В результате Release может остаться с **неполным** набором
ассетов. Это не тихая ошибка — запуск Actions красный и виден. Исправление:
перезапустить workflow (`workflow_dispatch` на том же теге) — загрузка перезапишет
ассеты начисто и добьёт недостающие.

## Бэкапы

В проекте нет пользовательских данных. Бэкапы не требуются. Исходный код
хранится на GitHub и локально у разработчика.
